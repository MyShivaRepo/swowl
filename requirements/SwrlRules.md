# Requirements — Tab "SwrlRules"

> Generated by reverse-engineering of the SWOWL source code · version 1.0 · 2026-06-06

---

## 1. Navigation et liste des règles

- **REQ-SWRL-001** — Display the list of SWRL rules in a dedicated left panel with title 'SWRL Rules'
  - *If* If the SwrlRules tab is active, Then the left panel displays the list of all SWRL rules loaded in memory.

- **REQ-SWRL-002** — Each rule is displayed with its label (or ID if no label) and, as subtext, its ID when a label is set
  - *If* If the rule has a label, Then the label is shown as main text and the ID as small subtext.

- **REQ-SWRL-003** — Rules referencing a deleted entity (class or property) are displayed in red in the list
  - *If* If an atom in the rule references a class or property not present in the ontology, Then the rule is shown in red in the list.

- **REQ-SWRL-004** — Clicking a rule in the list selects it and displays its edit form in the right panel
  - *If* If the user clicks a rule in the list, Then the rule is highlighted and its detail is rendered in the right panel.

- **REQ-SWRL-005** — The current selection is restored after a tab reload
  - *If* If the tab is reinitialized, Then the previously selected rule is automatically re-selected.

- **REQ-SWRL-006** — The list panel is horizontally resizable via a draggable split handle
  - *If* If the user drags the split handle, Then the left panel width is dynamically adjusted between 120 px and 400 px.

---

## 2. Recherche et filtrage

- **REQ-SWRL-007** — A search field at the bottom of the left panel filters rules in real time
  - *If* If the user types in the search field, Then the list is immediately filtered without reload.

- **REQ-SWRL-008** — Search covers the ID, label, comment and atom values (class_id, property_id, var, subject, object, value)

- **REQ-SWRL-009** — A ✕ button appears in the search field to clear the input and show all rules again
  - *If* If the search query is non-empty, Then a ✕ button appears; if the user clicks it, Then the full list is restored.

- **REQ-SWRL-010** — When no rule matches the search, a 'No matching rule' message is displayed
  - *If* If the filter returns no results, Then the list area displays 'No matching rule'.

---

## 3. Création et suppression de règles

- **REQ-SWRL-011** — A ➕ button in the list panel header creates a new empty SWRL rule
  - *If* If the user clicks ➕, Then a rule with a unique auto-generated ID (NewRule, NewRule1, …) is created via POST /api/swrl-rules and immediately selected.

- **REQ-SWRL-012** — The auto-generated ID increments a numeric suffix to ensure uniqueness (NewRule, NewRule1, NewRule2, …)

- **REQ-SWRL-013** — Each list row displays a delete button that removes the rule after confirmation
  - *If* If the user clicks the delete button, Then a confirmation dialog is shown; if confirmed, the rule is deleted via DELETE /api/swrl-rules/{id}.

- **REQ-SWRL-014** — A 404 error from the backend during deletion is silently ignored (idempotency)
  - *If* If the backend responds 404 during DELETE, Then no error message is shown and the rule is removed from the UI.

- **REQ-SWRL-015** — After deletion, the right panel displays an empty state inviting the user to select or create a rule

---

## 4. Édition des métadonnées de la règle

- **REQ-SWRL-016** — The edit form exposes three metadata fields: ID (required), Label and Comment

- **REQ-SWRL-017** — Typing in the ID field automatically replaces spaces with underscores
  - *If* If the user types a space in the ID field, Then the character is immediately replaced by '_'.

- **REQ-SWRL-018** — Any change to the ID, label or comment triggers automatic save via PUT /api/swrl-rules/{id}
  - *If* If the user leaves a metadata field (onchange event), Then the rule is automatically saved without explicit action.

- **REQ-SWRL-019** — Renaming a rule (ID change) is propagated: the old ID is used as the PUT key and the new ID is persisted
  - *If* If the rule ID is changed, Then the PUT call uses the old ID as the URL identifier and sends the new ID in the body.

- **REQ-SWRL-020** — The ID field receives focus automatically when the edit form of an existing rule is opened

- **REQ-SWRL-021** — The Comment field is a vertically resizable textarea

---

## 5. Structure des règles : atomes Body (if) et Head (then)

- **REQ-SWRL-022** — The rule is divided into two sections: 'if' (antecedents / body) and 'then' (consequents / head)

- **REQ-SWRL-023** — The 'if' section accepts four atom types: type_atom (Class), property_atom (Property), equality_atom (≟), naf_block (NAF)

- **REQ-SWRL-024** — The 'then' section accepts four atom types: type_atom, property_atom, equality_atom and conditional (If … Then)

- **REQ-SWRL-025** — A button per atom type is displayed in each section's toolbar to add an atom of that type
  - *If* If the user clicks an add button, Then a new atom of the corresponding type is appended to the list and the rule is saved.

- **REQ-SWRL-026** — A type_atom (Class) contains a variable field (?var) and a class selector with the syntax '?var is a [Class]'

- **REQ-SWRL-027** — A property_atom contains a subject field (?subj), a property selector and an object field (?obj)

- **REQ-SWRL-028** — An equality_atom contains a variable field, a comparison operator selector (=, !=, >, >=, <, <=) and a value field

- **REQ-SWRL-029** — A naf_block (Negation As Failure) is a nested block that can contain type_atom, property_atom and equality_atom atoms

- **REQ-SWRL-030** — A conditional (If … Then) in the head section is a nested block containing a condition list and a consequent list, each enrichable with atoms

- **REQ-SWRL-031** — Conditionals can be nested recursively (conditional inside a conditional's consequent)

- **REQ-SWRL-032** — Each atom has a ✕ button to remove it from its list, with automatic save
  - *If* If the user clicks an atom's ✕ button, Then the atom is removed from the list and the rule is immediately saved.

- **REQ-SWRL-033** — When a section is empty, a placeholder message is shown in italics ('— add antecedents —' or '— add consequents —')

---

## 6. Sélecteurs de classe, propriété et individu

- **REQ-SWRL-034** — A class selector opens as a fixed-positioned popup below or above the button depending on available space
  - *If* If the space below the button is insufficient, Then the popup opens above.

- **REQ-SWRL-035** — The class selector displays the ontology class tree; a click selects the class and saves the rule

- **REQ-SWRL-036** — On a type_atom with a class already selected, left-click navigates to that class in the Classes tab, right-click reopens the selector
  - *If* If the user right-clicks the class pill, Then the context menu is prevented and the class selector opens.

- **REQ-SWRL-037** — If the class referenced by an atom has been deleted, the pill shows '⚠ deleted' in red

- **REQ-SWRL-038** — The property selector lists object properties (blue dot) and datatype properties (green dot) sorted alphabetically with a separator

- **REQ-SWRL-039** — On a property_atom with a property already selected, left-click navigates to the property in its dedicated tab, right-click reopens the selector

- **REQ-SWRL-040** — The equality_atom offers a button to open a two-panel individual picker modal
  - *If* If the user clicks the individual picker button, Then a modal appears with the class list on the left and filtered individuals on the right.

- **REQ-SWRL-041** — The individual picker offers 'owl:Thing' to view all individuals, and each class with the corresponding individual count

- **REQ-SWRL-042** — In the individual picker, a double-click on an individual immediately confirms the selection
  - *If* If the user double-clicks an individual, Then the selection is confirmed and the modal closes without clicking OK.

- **REQ-SWRL-043** — If the individual selected in an equality_atom exists in the ontology, it is displayed as a navigable pill; otherwise as a free text field

- **REQ-SWRL-044** — All pickers (class, property) close automatically when the user clicks outside
  - *If* If the user clicks outside the popup, Then the picker is hidden and the click listener is removed.

---

## 7. Glisser-déposer (drag-and-drop)

- **REQ-SWRL-045** — Each atom in a list (body, head, atoms, condition, consequent) has a ⠿ handle allowing reordering via drag-and-drop

- **REQ-SWRL-046** — Drag-and-drop is restricted to the same parent list; an atom cannot be moved to a different list
  - *If* If the user drags an atom over an atom from a different list, Then the dragOver event is ignored (no dropEffect).

- **REQ-SWRL-047** — During drag, the target atom is highlighted via the CSS class 'swrl-drag-target'

- **REQ-SWRL-048** — After dropping, atoms are reordered in memory, the form is re-rendered and the rule is automatically saved
  - *If* If an atom is dropped at a new position, Then the list is updated and a PUT /api/swrl-rules/{id} is triggered.

---

## 8. Interactions backend (API REST)

- **REQ-SWRL-049** — GET /api/swrl-rules returns the list of all SWRL rules of the current ontology

- **REQ-SWRL-050** — POST /api/swrl-rules creates a new rule and returns 201; a rule with an already existing ID causes a 409 error
  - *If* If the submitted ID already exists, Then the backend responds 409 Conflict.

- **REQ-SWRL-051** — PUT /api/swrl-rules/{rule_id} updates an existing rule; if the ID changes, the uniqueness of the new ID is verified
  - *If* If the new ID is already assigned to another rule, Then the backend responds 409 Conflict.

- **REQ-SWRL-052** — DELETE /api/swrl-rules/{rule_id} deletes the rule; if it does not exist, the backend returns 404

- **REQ-SWRL-053** — Renaming a class in the ontology automatically propagates the change into the class_id of all SWRL atoms
  - *If* If a class is renamed, Then all type_atom atoms referencing that ID are cascaded-updated on the backend.

- **REQ-SWRL-054** — Renaming a property in the ontology automatically propagates the change into the property_id of all SWRL atoms
  - *If* If a property is renamed, Then all property_atom atoms referencing it are cascaded-updated on the backend.

- **REQ-SWRL-055** — SWRL rules are exportable in JSON format via the endpoint GET /api/export/{name}?fmt=swrl
  - *If* If the 'swrl' export format is requested, Then a JSON file containing all rules is returned with the extension _rules.swrl.json.

- **REQ-SWRL-056** — Any backend error is shown to the user via the UI.error notification
  - *If* If an API request fails, Then an error message is displayed and no local data is modified.
