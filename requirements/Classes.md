# Requirements — Tab "Classes"

> Generated by reverse-engineering of the SWOWL source code · version 1.0 · 2026-06-06

---

## 1. Navigation and selection in the class tree

- **REQ-CLS-001** — Class selection
  - *If* the user clicks a node in the tree, *Then* the class is highlighted, the right panel displays its form, and the "Super Classes" panel is updated.

- **REQ-CLS-002** — Selection of `owl:Thing` (root)
  - *If* the user clicks the root node, *Then* the right panel displays a welcome message with a "Create Class" button, and the "Super Classes" panel is cleared.

- **REQ-CLS-003** — Silent save on selection change
  - *If* a class is being edited and the user selects another class, *Then* the current class is silently saved before switching selection.

- **REQ-CLS-004** — Selection restored after re-render
  - *If* the section is re-rendered (e.g. after save), *Then* the previous selection is restored.

- **REQ-CLS-005** — Context-sensitive toolbar buttons
  - *If* `owl:Thing` is selected, *Then* only the "Child" button is enabled.
  - *If* a class is selected, *Then* "Child", "Sibling" and "Delete" buttons are all enabled.
  - *If* nothing is selected, *Then* all buttons are disabled.

---

## 2. Class hierarchy tree construction and display

- **REQ-CLS-010** — Hierarchy computation
  - *If* classes have a `subClassOf` pointing to other classes in the ontology, *Then* they are displayed as children (alphabetically sorted).
  - *If* a class has no internal parent, *Then* it is placed under `owl:Thing`.
  - *If* a class only has external prefixed parents (e.g. `skos:Concept`), *Then* it is displayed as an independent root.

- **REQ-CLS-011** — Expandable / collapsible nodes
  - *If* a class has children, *Then* a ▶ triangle is shown; clicking toggles the subtree open/closed state.

- **REQ-CLS-012** — Automatic ancestor expansion
  - *If* a class is selected via external navigation, *Then* all its ancestors are automatically expanded to make it visible.

- **REQ-CLS-013** — Empty state message
  - *If* no class exists in the ontology, *Then* a "No classes" message is displayed in the tree panel.

- **REQ-CLS-014** — Visual identifier
  - Each class is represented by a brown dot (`.cls-dot`). The `owl:Thing` root has a distinct visual style.

---

## 3. Class creation

- **REQ-CLS-020** — Create a child class
  - *If* the user clicks "Child" (toolbar or context menu), *Then* a new class is created with `subClassOf = [selected_parent_id]`, and the parent is expanded in the tree.

- **REQ-CLS-021** — Create a sibling class
  - *If* the user clicks "Sibling", *Then* a new class is created with the same parents as the current class.

- **REQ-CLS-022** — Automatic name generation
  - *If* a new class is created, *Then* its ID is `NewClass`, `NewClass1`, `NewClass2`… (first name not already in use).

- **REQ-CLS-023** — Initial structure
  - Every new class is created with: `id`, `annotations.labels=[]`, `annotations.comments=[]`, `subClassOf`, `equivalentClass=[]`, `disjointWith=[]`.

- **REQ-CLS-024** — API call on creation
  - *If* a class is created, *Then* `POST /api/classes` is called. If the ID already exists, the backend returns HTTP 409.

- **REQ-CLS-025** — ID uniqueness (backend)
  - *If* the submitted ID already exists, *Then* the backend raises `HTTP 409 "Classe 'X' already exists"`.

---

## 4. Class renaming

- **REQ-CLS-030** — Rename via ID field
  - *If* the user modifies the ID field and saves, *Then* `PUT /api/classes/{old_id}` is called with the new ID, and a confirmation message is displayed.

- **REQ-CLS-031** — Cascade propagation on rename (backend)
  - *If* a class is renamed, *Then* the backend propagates the change across: `subClassOf` / `equivalentClass` / `disjointWith` of all classes, `domain` / `range` of ObjectProperties, `domain` of DatatypeProperties, `types` of Individuals, and SWRL rule atoms.

- **REQ-CLS-032** — New ID uniqueness (backend)
  - *If* the new ID is already used by another class, *Then* the backend raises `HTTP 409`.

---

## 5. Class deletion

- **REQ-CLS-040** — Confirmation before deletion
  - *If* the user clicks "Delete", *Then* a confirmation dialog is displayed. If descendant classes exist, their count and IDs are listed.

- **REQ-CLS-041** — Blocked if individuals use the class
  - *If* individuals have the class to be deleted (or any of its descendants) as their `type`, *Then* deletion is blocked (frontend and backend) with a message listing the concerned individuals (max 3 shown).

- **REQ-CLS-042** — Cascade deletion of descendants (backend)
  - *If* a class is deleted, *Then* all its descendant classes (recursively) are also deleted, and these classes are removed from all property `domain` lists.

- **REQ-CLS-043** — Selection reset after deletion
  - After successful deletion, the selection is cleared and the tree is refreshed.

---

## 6. Drag & Drop

- **REQ-CLS-050** — Drag start
  - *If* the user starts dragging a node, *Then* the class ID is stored, and the dragged class receives a `dragging` style (reduced opacity).

- **REQ-CLS-051** — Visual indicator on target
  - *If* the mouse hovers over a valid node during drag, *Then* the target receives a `drag-over` style.

- **REQ-CLS-052** — Drop on descendant forbidden
  - *If* the drop target is a descendant of the dragged class, *Then* the drop is rejected with a "Cannot drop on a descendant — would create a cycle" message.

- **REQ-CLS-053** — Move by drop
  - *If* a drop is performed on a valid target, *Then* the dragged class's parents are replaced by `[targetId]` (or `[]` if dropped on `owl:Thing`). Object restrictions in `subClassOf` are preserved. `PUT /api/classes/{id}` is called and a "Moved" message is shown.

- **REQ-CLS-054** — Post-drag cleanup
  - After drag end, `dragging` and `drag-over` styles are removed from all elements.

---

## 7. Context menu (right-click)

- **REQ-CLS-060** — Context menu display
  - *If* the user right-clicks a node, *Then* a floating menu is displayed at cursor position.

- **REQ-CLS-061** — Menu content by target
  - *If* right-clicking on `owl:Thing`, *Then* the menu contains only "Add Child Class".
  - *If* right-clicking on a class, *Then* the menu contains "Add Child Class", "Add Sibling Class", and "Delete" (in red).

- **REQ-CLS-062** — Adaptive positioning
  - *If* the menu would overflow the screen edge, *Then* it is repositioned to remain visible.

- **REQ-CLS-063** — Auto-close
  - *If* the user clicks outside the menu, *Then* the menu is closed.

---

## 8. "Super Classes" panel

- **REQ-CLS-070** — Display with ancestor chain
  - *If* a class is selected, *Then* the panel displays each direct parent with its full ancestor chain up to `owl:Thing` (indirect ancestors shown with reduced opacity).

- **REQ-CLS-071** — Navigate to an ancestor
  - *If* the user clicks an ancestor name in the panel, *Then* that class is selected in the tree.

- **REQ-CLS-072** — Navigate to `owl:Thing`
  - *If* the user clicks `owl:Thing` in the panel, *Then* the root is selected.

- **REQ-CLS-073** — Remove a direct parent
  - *If* the user clicks ✕ of a direct parent, *Then* that parent is removed and the class is auto-saved.

- **REQ-CLS-074** — Add a super-class
  - *If* the user clicks ＋ in the panel, *Then* a dropdown lists available classes (excluding the current class and already listed parents). Selection triggers auto-save.

- **REQ-CLS-075** — `owl:Thing` placeholder
  - *If* a class has no parent, *Then* `owl:Thing` is displayed in italics as the sole ancestor in the panel.

---

## 9. Annotations

- **REQ-CLS-080** — Display existing annotations
  - *If* a class has annotations (`rdfs:label`, `rdfs:comment`, others), *Then* they are displayed in a table with columns: Property / Value / Language.

- **REQ-CLS-081** — Add `rdfs:label`
  - *If* the user clicks "+ label", *Then* a new row is added with property `rdfs:label` and the default language.

- **REQ-CLS-082** — Add `rdfs:comment`
  - *If* the user clicks "+ comment", *Then* a new row is added with `rdfs:comment`.

- **REQ-CLS-083** — Add an arbitrary annotation property
  - *If* the user clicks "+ Annotation Property", *Then* a picker lists built-in OWL 2 properties and user-defined annotation properties. Selection adds a row to the table.

- **REQ-CLS-084** — Remove an annotation
  - *If* the user clicks ✕ of an annotation row, *Then* the row is deleted and auto-save is triggered.

- **REQ-CLS-085** — Language field per annotation
  - Each annotation has a language field (free text) with a ▼ button opening a language selector.

- **REQ-CLS-086** — Navigate to annotation property
  - *If* the user clicks the name of an annotation property, *Then* the application navigates to that property in the AnnotationProperties tab.

- **REQ-CLS-087** — Annotation collection on save
  - Only annotations with a non-empty value are persisted. Format: `{value, lang}` for labels/comments, `{property, value}` for others.

---

## 10. Restrictions and properties (Asserted / Inherited)

- **REQ-CLS-090** — "Asserted Properties" section
  - *If* a class has restrictions in `subClassOf`, *Then* they are grouped by property with a range/multiplicity chip if the property has a `range` defined.

- **REQ-CLS-091** — "Inherited Properties" section
  - *If* a class has super-classes with restrictions, *Then* these are displayed read-only with the origin class indicated (↑ ClassName). The section is collapsible.

- **REQ-CLS-092** — Add an existing property
  - *If* the user clicks "+ property", *Then* a picker lists all ObjectProperties and DatatypeProperties not yet asserted. Selection adds the property as an empty group.

- **REQ-CLS-093** — Create ObjectProperty with current class as domain
  - *If* the user clicks the "OP" button, *Then* a new ObjectProperty is created with `domain = [current_class_id]`, and the application navigates to the ObjectProperties tab.

- **REQ-CLS-094** — Create DatatypeProperty with current class as domain
  - *If* the user clicks the "DT" button, *Then* a new DatatypeProperty is created with `domain = [current_class_id]`, and the application navigates to the DatatypeProperties tab.

- **REQ-CLS-095** — Add a restriction to a property
  - *If* the user right-clicks an asserted property, *Then* a menu offers 6 restriction types: `∃ someValuesFrom`, `∀ allValuesFrom`, `∋ hasValue`, `= exactCardinality`, `≥ minCardinality`, `≤ maxCardinality`.

- **REQ-CLS-096** — Filler / cardinality toggle
  - *If* the restriction type contains "Cardinality", *Then* the filler field is hidden and a numeric field is shown. Otherwise, the class filler selector is shown.

- **REQ-CLS-097** — Filler class selection
  - *If* the user clicks the filler button, *Then* a dropdown displays the full class hierarchy (including `owl:Thing`).

- **REQ-CLS-098** — Navigation from an inherited restriction
  - *If* the filler of an inherited restriction is a class in the ontology, *Then* its name is clickable and navigates to that class.

- **REQ-CLS-099** — Delete a child restriction
  - *If* the user clicks ✕ of a restriction child, *Then* the node is deleted and auto-save is triggered.

- **REQ-CLS-100** — Delete an entire property
  - *If* the user clicks ✕ of a property group, *Then* the property and all its restrictions are removed, and auto-save is triggered.

- **REQ-CLS-101** — Presence marker (`_marker`)
  - *If* a property is added with no child restrictions, *Then* it is persisted via a marker `{type: '_marker', property: propId}` to maintain the `domain ↔ class` link.

- **REQ-CLS-102** — Multiplicity chip
  - *If* an asserted property has a `range` defined, *Then* a chip `(single RangeName)` or `(multiple RangeName)` is displayed depending on whether the property is functional.

- **REQ-CLS-103** — Navigate to property
  - *If* the user clicks the name of a property (asserted or inherited), *Then* the application navigates to the corresponding tab (ObjectProperties or DatatypeProperties).

- **REQ-CLS-104** — `_marker` ↔ domain sync (backend)
  - *If* a class is updated, *Then* the backend synchronises property `domain` lists based on added or removed `_marker` entries.

---

## 11. Equivalent classes

- **REQ-CLS-110** — Display
  - String-typed equivalent classes are displayed in the "Equivalent" frame of the form.

- **REQ-CLS-111** — Add an equivalent class
  - *If* the user clicks ＋ in the "Equivalent" frame, *Then* a tree picker is shown (excluding the current class and already listed equivalents). Selection triggers auto-save.

- **REQ-CLS-112** — Navigate to an equivalent class
  - *If* the user clicks the name of an equivalent class, *Then* the application navigates to that class.

- **REQ-CLS-113** — Remove an equivalence
  - *If* the user clicks ✕ of an equivalence, *Then* the item is removed and auto-save is triggered.

- **REQ-CLS-114** — Automatic symmetry (backend)
  - *If* class A is defined as equivalent to B, *Then* the backend automatically adds A as equivalent to B. If the equivalence is removed, the symmetry is also removed.

---

## 12. Disjoint classes

- **REQ-CLS-120** — Display
  - Disjoint classes are displayed in the "Disjoints" frame of the form.

- **REQ-CLS-121** — Add a disjoint class
  - *If* the user clicks ＋ in the "Disjoints" frame, *Then* a tree picker is shown (excluding the current class and already listed disjoints). Selection triggers auto-save.

- **REQ-CLS-122** — Remove a disjoint
  - *If* the user clicks ✕ of a disjoint, *Then* the item is removed and auto-save is triggered.

- **REQ-CLS-123** — Automatic symmetry (backend)
  - *If* class A is declared disjoint from B, *Then* the backend automatically adds A to B's disjoints, and removes the symmetry if the disjunction is deleted.

---

## 13. OWL identifier validation (NCName)

- **REQ-CLS-130** — Real-time sanitization (frontend)
  - *If* the user types in the ID field, *Then* spaces are replaced by `_` and leading digits are removed.

- **REQ-CLS-131** — Validation before save (frontend)
  - *If* the ID is empty, *Then* the message "ID is required." is shown.
  - *If* the ID starts with a digit, *Then* the message "Identifier cannot start with a digit (OWL NCName rule)." is shown.

- **REQ-CLS-132** — Backend validation
  - *If* the received ID is empty or starts with a digit, *Then* the backend raises `HTTP 422`.

---

## 14. Auto-save

- **REQ-CLS-140** — Auto-save trigger
  - *If* a class is being edited and the user modifies any field (annotation, super-class, equivalent, disjoint, restriction), *Then* `autoSave()` is triggered automatically.

- **REQ-CLS-141** — Silent save
  - *If* the user switches class without an explicit save, *Then* the current state is persisted via `PUT /api/classes/{id}` without re-rendering or showing a message.

- **REQ-CLS-142** — Full collection on save
  - The save operation collects from the DOM: ID, annotations, super-classes, equivalences, disjoints, and restrictions.

---

## 15. Cross-navigation

- **REQ-CLS-150** — Full IRI displayed
  - *If* the ontology has a base ID, *Then* the full IRI of the class is displayed as `{baseIri}#{classId}`.

- **REQ-CLS-151** — Navigate to referenced entity
  - *If* the user clicks a super-class, an equivalent class, or a restriction filler, *Then* the application navigates to that class.

- **REQ-CLS-152** — "Where Used in Rules" frame
  - *If* the class is referenced in at least one SWRL rule, *Then* a "Where Used in Rules" frame is displayed at the bottom of the form, with clickable rules navigating to the SWRL Rules tab.

---

## 16. REST API interface (backend)

| Operation | Method | Endpoint |
|---|---|---|
| List classes | GET | `/api/classes` |
| Create a class | POST | `/api/classes` |
| Get a class | GET | `/api/classes/{id}` |
| Update / rename | PUT | `/api/classes/{id}` |
| Delete | DELETE | `/api/classes/{id}` |

- **REQ-CLS-160** — Immediate persistence
  - Every change (create, update, delete) triggers a disk save on the backend.

---

## 17. Resizable layout

- **REQ-CLS-170** — Horizontal resize: tree / form
  - A vertical splitter allows resizing the tree panel width (between 160 px and 520 px) by horizontal drag.

- **REQ-CLS-171** — Vertical resize: tree ↕ Super Classes
  - A horizontal splitter allows resizing the height between the "Asserted Hierarchy" sub-panel and the "Super Classes" sub-panel.

- **REQ-CLS-172** — Form section resizing
  - Horizontal splitters allow resizing the Annotations / Restrictions / Disjoints-Equivalents sections in the form panel.
