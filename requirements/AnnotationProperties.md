# Requirements — Tab "AnnotationProperties"

> Generated by reverse-engineering of the SWOWL source code · version 1.0 · 2026-06-06

---

## 1. Affichage général et mise en page

- **REQ-AP-001** — The tab displays a split panel with two columns separated by a draggable vertical resizer.
  - *If* If the user drags the vertical split handle, then the tree panel width is recalculated between 160 px and 520 px.

- **REQ-AP-002** — The left panel is itself divided into two vertically resizable zones: the tree zone (top) and the Super Properties zone (bottom).
  - *If* If the user drags the internal horizontal resizer, then the relative heights of the two left-panel zones are adjusted.

- **REQ-AP-003** — When no property is selected, the detail panel displays a prompt message and a '＋ Create Annotation Property' button.

- **REQ-AP-004** — Built-in properties are displayed only if the ontology is OWL-based (root class owl:Thing).
  - *If* If the ontology root is not owl:Thing, then the rdfs: and owl: namespace nodes and their built-in properties are not displayed.

---

## 2. Arbre de navigation des propriétés

- **REQ-AP-005** — The tree displays two namespace root nodes (rdfs: and owl:) representing the built-in property namespaces.

- **REQ-AP-006** — Under each namespace node, built-in properties are listed read-only with their identifier in monospace font and a 'built-in' badge.

- **REQ-AP-007** — User-defined properties without a parent and without a known namespace prefix are displayed as orphan nodes at the tree root.

- **REQ-AP-008** — Tree nodes are sorted alphabetically (case-insensitive) at each level.

- **REQ-AP-009** — A node with children displays a clickable chevron (▶) to expand/collapse; a leaf node displays a bullet marker (◦).
  - *If* If the user clicks a node's chevron, then that node's expanded/collapsed state is toggled and the tree is re-rendered.

- **REQ-AP-010** — The rdfs: and owl: nodes are expanded by default when the tab loads.

- **REQ-AP-011** — The selected node is visually highlighted by the CSS class 'selected'.

- **REQ-AP-012** — Node indentation in the tree is proportional to depth (depth × 16 + 6 px).

---

## 3. Sélection et panneau de détail

- **REQ-AP-013** — Clicking a namespace root node (rdfs: or owl:) displays in the detail panel the list of all its built-in properties with their comments.

- **REQ-AP-014** — Clicking a built-in property displays its identifier, its comment and the label 'Built-in OWL 2 annotation property — read-only.'; no field is editable.

- **REQ-AP-015** — Clicking a user-defined property displays the edit form with the ID field, the computed full IRI, and the annotations table.

- **REQ-AP-016** — The full IRI of the property is displayed as '{baseIRI}#{id}' and is not editable.
  - *If* If the ontology base IRI is empty, then the full IRI display is omitted.

---

## 4. Création de propriétés

- **REQ-AP-017** — The 'Child' button creates a new sub-property of the currently selected property.
  - *If* If a user or built-in property is selected, then the new property is created with subPropertyOf = [selectedId]; if a root namespace node is selected, then subPropertyOf is empty.

- **REQ-AP-018** — The 'Sibling' button creates a new property at the same level as the selected property, inheriting the same parents.
  - *If* If no user property is selected (root or built-in), then the Sibling button is hidden.

- **REQ-AP-019** — The name of the new property is automatically generated as 'NewAnnotationProperty', then 'NewAnnotationProperty1', etc., to avoid duplicates.
  - *If* If 'NewAnnotationProperty' already exists, then an incrementing numeric suffix is appended until a unique identifier is found.

- **REQ-AP-020** — After creation, the parent node is automatically expanded and the new property is selected in the tree.

- **REQ-AP-021** — Creating a property calls the POST /api/annotation-properties endpoint with the new property's JSON payload.
  - *If* If the identifier already exists server-side (HTTP 409), then an error is displayed and the creation is cancelled.

---

## 5. Édition et sauvegarde

- **REQ-AP-022** — The property identifier is editable via a text field; input is sanitized in real-time (spaces converted to '_', leading digits removed).
  - *If* If the user types a space or leading digit in the identifier field, then the field is automatically corrected without notification.

- **REQ-AP-023** — Saving is triggered automatically (auto-save) on each form change (onchange event).
  - *If* If the ID field is empty, then auto-save is cancelled without a network call.

- **REQ-AP-024** — If the identifier is changed, auto-save renames the property via PUT /api/annotation-properties/{old_id} and displays a success message 'Renamed → ...'.
  - *If* If renaming causes a conflict (HTTP 409), then an error message is displayed and the current selection is preserved.

- **REQ-AP-025** — After each successful save, the application state is refreshed (APP.refresh) and the current selection is restored in the tree.

- **REQ-AP-026** — The subPropertyOf relation is preserved during save from the current state store; it is not modifiable via the edit form.

- **REQ-AP-027** — The form allows adding annotations of type 'label', 'comment', and 'other' via a table of dynamic rows.
  - *If* If the user modifies a value in the annotation table, then auto-save is triggered.

- **REQ-AP-028** — ID validation rejects an empty ID or one starting with a digit (OWL NCName rule) before manual save.
  - *If* If the ID is invalid during an explicit save(), then a warning is displayed and the save is blocked.

---

## 6. Suppression de propriétés

- **REQ-AP-029** — The 'Delete' button (trash icon, danger style) deletes the selected user-defined property after confirmation.
  - *If* If the user cancels the confirmation dialog, then no deletion is performed.

- **REQ-AP-030** — The Delete button is hidden if a namespace root node or a built-in property is selected.
  - *If* If the selection is a built-in property or a root node, then the Delete button is hidden (display:none).

- **REQ-AP-031** — After successful deletion, the selection is reset, the detail panel returns to its empty state, and the tree is re-rendered.

- **REQ-AP-032** — Deletion calls the DELETE /api/annotation-properties/{prop_id} endpoint.
  - *If* If the property does not exist server-side (HTTP 404), then an error message is displayed.

---

## 7. Drag & Drop

- **REQ-AP-033** — User-defined properties are draggable in the tree to change their parent relationship via drag-and-drop.
  - *If* If a property is dragged, then it receives the CSS class 'dragging' and the drop target receives the CSS class 'drag-over' on hover.

- **REQ-AP-034** — A node cannot be dropped onto itself or onto any of its own descendants (cycle protection).
  - *If* If the drop target is a descendant of the dragged node, then the drop event is ignored (preventDefault not called).

- **REQ-AP-035** — Dropping a property onto a namespace root node (rdfs: or owl:) places it without a parent (orphan under that namespace).
  - *If* If the drop target is rdfs: or owl:, then subPropertyOf is reset to [] for the dragged property.

- **REQ-AP-036** — Dropping a property onto another property (user or built-in) sets it as a sub-property of the target (subPropertyOf = [targetId]).
  - *If* If the drop succeeds, then a success message '{id} moved' is displayed and the tree is re-rendered.

- **REQ-AP-037** — At the end of a drag (dragend), the CSS classes 'dragging' and 'drag-over' are removed from all relevant elements.

- **REQ-AP-038** — A drag-and-drop move calls the PUT /api/annotation-properties/{prop_id} endpoint with the new subPropertyOf value.

---

## 8. Panneau Super Properties

- **REQ-AP-039** — The 'Super Properties' panel displays the full ancestor chain of the selected property, from the direct parent up to the namespace root node.
  - *If* If the property has no parent, then the panel displays the message '— no super-property —'.

- **REQ-AP-040** — Each element in the ancestor chain is clickable and navigates to the corresponding property in the tree.
  - *If* If the user clicks an ancestor in the Super Properties panel, then APP.navigateTo('annotation-properties', id) is called.

- **REQ-AP-041** — Ancestors are displayed with increasing indentation and decreasing opacity for more distant levels.

- **REQ-AP-042** — When a namespace root node is selected, the Super Properties panel displays the message '— select a property —'.

---

## 9. Interactions backend (API REST)

- **REQ-AP-043** — The tab loads the property list via GET /api/annotation-properties when the application state is refreshed.

- **REQ-AP-044** — The data model of an annotation property includes: id (str), subPropertyOf (list[str]), comment (str), annotations (labels, comments, other).

- **REQ-AP-045** — The backend rejects creation or renaming if the identifier already exists (HTTP 409 Conflict).

- **REQ-AP-046** — The backend returns HTTP 404 if a property not found is requested (GET, PUT, DELETE).

---

## 10. Navigation croisée et intégration

- **REQ-AP-047** — Navigation links (nav-link) from other tabs (Classes, ObjectProperties, DatatypeProperties, Individuals) allow direct navigation to a specific annotation property.
  - *If* If APP.navigateTo('annotation-properties', id) is called, then the Annotation Properties tab is activated and property id is selected.

- **REQ-AP-048** — Built-in properties (rdfs:label, rdfs:comment, etc.) and user-defined properties are available in the annotation pickers of other tabs.

- **REQ-AP-049** — The current selection in the tree is restored after each re-render of the section (restoreSelection).
  - *If* If _selectedId is non-null during re-render, then selectProp(_selectedId) is called automatically.
