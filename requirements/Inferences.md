# Requirements — Tab "Inferences"

> Generated by reverse-engineering of the SWOWL source code · version 1.0 · 2026-06-06

---

## 1. Navigation et affichage général

- **REQ-INF-001** — The Inferences tab is optional and can be hidden/shown in the navigation
  - *If* If the tab is listed in APP._optional, then the user can toggle its visibility

- **REQ-INF-002** — The tab displays the title '🧠 Real-time Inferences' and a global recalculate button
  - *If* If the tab is active, then the header contains the label and a '↻ Recalculate' button

- **REQ-INF-003** — A 'Loading…' message is displayed in the panel during initial loading
  - *If* If navigation to the tab is triggered, then the panel shows 'Loading…' before the API response

- **REQ-INF-004** — If an API error occurs, an error message is displayed in the inference panel
  - *If* If the call to /api/inferences fails, then the panel displays the corresponding error message

- **REQ-INF-005** — The tab automatically triggers an inference recalculation on activation
  - *If* If the current section switches to 'inferences', then InferenceUI.refresh() is called immediately

## 2. Rafraîchissement automatique et manuel

- **REQ-INF-006** — The panel auto-refreshes every 3 seconds when the tab is active
  - *If* If the Inferences tab is the current section, then InferenceUI.refresh() is called every 3000 ms

- **REQ-INF-007** — Auto-refresh only triggers if the Inferences tab is currently active
  - *If* If APP.currentSection is not 'inferences', then the interval does not trigger any API call

- **REQ-INF-008** — The auto-refresh is cleanly stopped before starting a new one
  - *If* If startAutoRefresh() is called, then stopAutoRefresh() is first called to cancel the existing interval

- **REQ-INF-009** — A '↻' button in the panel header allows an immediate manual recalculation
  - *If* If the user clicks the '↻' button, then InferenceUI.refresh() is called

## 3. Indicateur de cohérence et violations

- **REQ-INF-010** — The panel header displays a green badge '🟢 Consistent' if no errors are detected
  - *If* If the error count is zero, then the badge shows 'badge-ok' with a green icon

- **REQ-INF-011** — The header displays a red badge '🔴 N error(s)' if errors are present
  - *If* If the error count is greater than zero, then the badge shows 'badge-error' with the counter

- **REQ-INF-012** — An orange badge '⚠️ N warning(s)' is displayed if warnings are present
  - *If* If the warning count is greater than zero, then a 'badge-warn' badge is added to the header

- **REQ-INF-013** — The Violations section displays each violation with its severity icon, entity and message
  - *If* If violations exist, then each row shows '🔴' for 'error' or '🟡' for 'warning', the entity code and the message text

- **REQ-INF-014** — If no violations are detected, the Violations section displays '✅ No violations detected.'
  - *If* If the violations list is empty, then the section displays the corresponding empty message

- **REQ-INF-015** — The backend detects disjointness violations between classes shared by an individual
  - *If* If an individual is declared of type T1 and T2 with T1 disjointWith T2, then an 'error' violation is generated

- **REQ-INF-016** — The backend detects domain consistency violations on object properties
  - *If* If an individual uses a property whose domain is D without being of type D, then a 'warning' violation is generated

- **REQ-INF-017** — The backend detects exact cardinality violations on classes
  - *If* If an individual of type C has a different number of values than the declared exact cardinality, then a 'warning' violation is generated

- **REQ-INF-018** — The backend detects functional property violations
  - *If* If an individual has more than one value for a functional property, then an 'error' violation is generated

- **REQ-INF-019** — The backend detects asymmetric property violations
  - *If* If prop(A,B) and prop(B,A) coexist for an asymmetric property, then an 'error' violation is generated

- **REQ-INF-020** — The backend detects irreflexive property violations
  - *If* If prop(A,A) is declared for an irreflexive property, then an 'error' violation is generated

- **REQ-INF-021** — The backend detects sameAs/differentFrom contradictions on individuals
  - *If* If an individual is both sameAs and differentFrom another individual, then an 'error' violation is generated

## 4. Sections collapsibles et tableaux d'inférences

- **REQ-INF-022** — Non-empty sections are collapsible by clicking on their title
  - *If* If the user clicks on a collapsible section title, then the 'open' class is toggled on the parent element

- **REQ-INF-023** — A '▶' visual indicator in the title signals collapsible sections
  - *If* If a section is collapsible, then a span.caret element containing '▶' is displayed in the title

- **REQ-INF-024** — The 'Resolved hierarchy' section displays the transitive closure of subClassOf as a table
  - *If* If classes have ancestors, then the table lists each class and all its ancestors with a visual tag

- **REQ-INF-025** — The 'Inherited restrictions' section displays restrictions inherited by each class
  - *If* If inherited restrictions exist, then the table lists the class, the source class and the restriction description

- **REQ-INF-026** — The 'Inferred types' section displays types inferred via property domains and ranges
  - *If* If inferred types exist, then the table lists the individual (with its label if available), the inferred type and the reason

- **REQ-INF-027** — The 'Inferred symmetric assertions' section displays deduced symmetric assertions
  - *If* If symmetric assertions exist, then the table lists individual, property, target and reason

- **REQ-INF-028** — The 'Inferred transitive assertions' section displays deduced transitive assertions
  - *If* If transitive assertions exist, then the table lists individual, property, target and reason

- **REQ-INF-029** — The 'Assertions (chains + inverses)' section groups transitive, reflexive, chain and inverse assertions
  - *If* If the engine produces combined assertions, then all are displayed in this section with individual, property, target and reason

- **REQ-INF-030** — The 'Inferred inverse restrictions' section displays class-level inferred inverse restrictions
  - *If* If inverse restrictions are computed, then the table lists the class, the inferred restriction (∃prop.filler) and the reason

- **REQ-INF-031** — The 'Inferred inverse properties' section displays owl:inverseOf symmetries not explicitly declared
  - *If* If A inverseOf B is declared but not B inverseOf A, then property B is listed with its reason

- **REQ-INF-032** — If a section is empty, a 'No [entity] detected/inferred.' message is displayed instead of the table
  - *If* If a section's result list is empty, then the section displays a paragraph with class='empty'

- **REQ-INF-033** — Empty sections with zero results are not rendered (symmetric, transitive and chain assertions)
  - *If* If the assertions list is empty, then _renderAssertions returns an empty string and the section does not appear

## 6. Moteur d'inférences backend

- **REQ-INF-040** — The engine computes the full transitive closure of subClassOf for all classes
  - *If* If class A is a subclass of B and B is a subclass of C, then A has C in its computed ancestors

- **REQ-INF-041** — The engine treats class equivalences as bilateral subclass relationships
  - *If* If A equivalentClass B is declared, then A ⊑ B and B ⊑ A are added to the direct superclasses

- **REQ-INF-042** — The engine infers individual types via the domain and range of object properties
  - *If* If an individual uses a property with domain D, it is inferred to be of type D; if an individual is the object of a property with range R, it is inferred to be of type R

- **REQ-INF-043** — The engine computes symmetric assertions for all properties declared symmetric
  - *If* If prop(A,B) and prop is symmetric, then the assertion prop(B,A) is inferred

- **REQ-INF-044** — The engine computes the transitive closure for all properties declared transitive
  - *If* If prop(A,B) and prop(B,C) and prop is transitive, then prop(A,C) is inferred

- **REQ-INF-045** — The engine generates reflexive assertions for all properties declared reflexive
  - *If* If prop is reflexive, then prop(A,A) is inferred for each individual A

- **REQ-INF-046** — The engine follows property chains (propertyChainAxiom) of arbitrary length
  - *If* If p1 ∘ p2 → prop is declared and p1(A,B) and p2(B,C), then prop(A,C) is inferred

- **REQ-INF-047** — The engine computes inverse assertions for all properties with a declared inverseOf
  - *If* If prop(A,B) and invProp = inverseOf(prop), then invProp(B,A) is inferred

- **REQ-INF-048** — The engine infers owl:inverseOf symmetry: if A inverseOf B is declared, B inverseOf A is inferred
  - *If* If prop_A inverseOf prop_B is declared and prop_B inverseOf prop_A is not, then the inverse inference is added

- **REQ-INF-049** — The engine infers class-level inverse restrictions from someValuesFrom/allValuesFrom restrictions and inverseOf
  - *If* If C ⊑ ∃prop.D and prop inverseOf invProp, then D ⊑ ∃invProp.C is inferred

- **REQ-INF-050** — The backend raises a 404 error if no ontology is loaded when computing inferences
  - *If* If no ontology is present in memory, then the API returns HTTP 404 with the corresponding message

- **REQ-INF-051** — The backend exposes a dedicated /api/inferences/violations endpoint returning only violations
  - *If* If /api/inferences/violations is called, then only the violations list from the full result is returned

- **REQ-INF-052** — The backend exposes a dedicated /api/inferences/subclass-closure endpoint returning only the transitive closure
  - *If* If /api/inferences/subclass-closure is called, then only the subClassOf closure dictionary is returned

- **REQ-INF-053** — Transitive closure results are sorted alphabetically in the API response
  - *If* If the closure is returned, then the ancestor lists are sorted via sorted()
