# Requirements — Tab "Individuals"

> Generated by reverse-engineering of the SWOWL source code · version 1.0 · 2026-06-06

---

## 1. Disposition et navigation (3 colonnes)

- **REQ-IND-001** — The tab displays a 3-column resizable layout: class tree (col. 1), individual list (col. 2), detail form (col. 3).
  - *If* If the Individuals tab is open Then the 3 panels are rendered simultaneously in a section-split container.

- **REQ-IND-002** — Both column dividers are mouse-draggable to adjust each panel's width.
  - *If* If the user drags a divider Then the adjacent panel width is recalculated between defined min/max bounds (120–520 px for col. 1, 100–400 px for col. 2).

- **REQ-IND-003** — Column 1 displays the OWL class tree with owl:Thing as the universal root and an individual count per node.
  - *If* If a class has individuals (direct or inherited) Then its count is shown to the right of its label.

- **REQ-IND-004** — The individual count on a class node is transitive: it includes individuals from all subclasses.

- **REQ-IND-005** — Selecting a class filters the individual list (col. 2) transitively and updates column 2's title.
  - *If* If the user clicks a class Then the list shows only individuals whose type belongs to that class or any of its subclasses.

- **REQ-IND-006** — Selecting owl:Thing displays all individuals without filtering.
  - *If* If owl:Thing is selected Then all individuals of the ontology are listed.

- **REQ-IND-007** — The individual list (col. 2) is sorted alphabetically by the resolved display label, or by ID if no label.

- **REQ-IND-008** — The selection and chosen class are restored after an application refresh.
  - *If* If APP.refresh() is called Then IndividualEditor.restoreSelection() restores the highlighting of the selected class and individual.

---

## 2. Création d'un individu

- **REQ-IND-009** — A ➕ button in column 2's header opens a blank individual creation form in column 3.
  - *If* If the user clicks ➕ Then a ghost placeholder is prepended to the list and a blank form is shown in col. 3.

- **REQ-IND-010** — When the creation form opens, an identifier is pre-generated by Settings.generateIndividualId and the field is auto-focused and selected.
  - *If* If the creation form is displayed Then the ind-id input is focused and its content selected within 30 ms.

- **REQ-IND-011** — Saving a new individual is triggered automatically when the ID field loses focus, if the field is not empty.
  - *If* If the user leaves the ind-id field (onblur) and the value is not empty Then IndividualEditor.save(true) is called.

- **REQ-IND-012** — The identifier is validated client-side (NCName rule) before submission; spaces are converted to underscores on both frontend and backend.
  - *If* If the ID contains spaces Then they are replaced by '_' before the POST/PUT request.

- **REQ-IND-013** — If an individual with the same ID already exists, the backend returns HTTP 409 and an error message is displayed.
  - *If* If POST /api/individuals receives an already-existing ID Then HTTP 409 is returned with an explicit message.

- **REQ-IND-014** — A Cancel button on the creation form discards the input and resets col. 3 to its empty state.
  - *If* If the user clicks Cancel Then the ghost placeholder is removed, selection is cleared and col. 3 shows the empty screen.

- **REQ-IND-015** — If a class is selected when creating, it is pre-filled in the Types field of the form.
  - *If* If _selectedClassId is non-null when newIndividual() is called Then types: [selectedClassId] is included in the creation object.

---

## 3. Édition et auto-sauvegarde

- **REQ-IND-016** — Selecting an existing individual loads its detail form in col. 3 with all its data.

- **REQ-IND-017** — Any change to a form field triggers an immediate auto-save (onchange → autoSave → PUT /api/individuals/{id}).
  - *If* If _editingId is non-null and a field changes Then autoSave() calls save(false) without user confirmation.

- **REQ-IND-018** — Renaming an individual (changing the ID) updates all references in other individuals (objectAssertions, sameAs, differentFrom) on the backend.
  - *If* If an individual's ID changes via PUT /api/individuals/{ind_id} Then all occurrences of the old ID in other individuals are replaced with the new ID.

- **REQ-IND-019** — The edit form displays the individual's full IRI (baseIri#localId) for informational purposes.
  - *If* If the ontology has a base ID Then the full IRI is displayed below the form header.

- **REQ-IND-020** — The user can add, edit, or delete rdfs:label, rdfs:comment annotations and custom annotation properties.
  - *If* If an annotation row is added or removed on an existing individual Then autoSave() is triggered.

- **REQ-IND-021** — The user can add or remove types (rdf:type) from a class picker embedded in the form.
  - *If* If a type is added or removed on an existing individual Then autoSave() is triggered.

- **REQ-IND-022** — The form dynamically shows property panels (ObjectProperty and DatatypeProperty) derived from the individual's types, separated into inherited and direct properties.
  - *If* If the individual has types Then properties are collected via _getClassProperties and rendered in order: inherited first (ascending depth then alpha), direct afterwards.

- **REQ-IND-023** — For a functional property, the ➕ add button is hidden once a value is present.
  - *If* If data-single='true' and at least one ind-prop-row exists Then the ind-prop-add-{safeId} button is hidden.

- **REQ-IND-024** — For DataProperty values, if the value starts with http:// or https://, a clickable link button is displayed next to the field.
  - *If* If a DataProperty value is an HTTP(S) URL Then a 🔗 icon with target='_blank' is displayed.

- **REQ-IND-025** — Property panels within the form are vertically resizable via h-resizer dividers.

- **REQ-IND-026** — The form displays a 'Where used' block showing SWRL rules that reference this individual.
  - *If* If the individual is referenced in SWRL rules Then _whereUsedFrame() returns a panel listing those rules.

## 4. Suppression

- **REQ-IND-027** — Deleting an individual requires confirmation via a dialog before sending DELETE /api/individuals/{id}.
  - *If* If the user clicks the delete button Then UI.confirm() is displayed; deletion only occurs if confirmed.

- **REQ-IND-028** — Deleting an individual automatically removes inverse assertions (owl:inverseOf) pointing to it in other individuals.
  - *If* If DELETE /api/individuals/{id} is called Then _sync_inverse_assertions() removes all inverse ObjectPropertyAssertions linked to that individual.

- **REQ-IND-029** — Multi-selection deletion sequentially deletes all selected individuals with a single confirmation.
  - *If* If N individuals are selected and the user confirms Then each individual is deleted via API.deleteIndividual() in a for…of loop.

- **REQ-IND-030** — After deletion, columns 1, 2, and 3 are refreshed and the delete button is disabled.

---

## 5. Sélection et multi-sélection

- **REQ-IND-031** — A single click on an individual selects it, loads its form in col. 3, and sets the Shift+Click anchor.

- **REQ-IND-032** — A Shift+Click selects a range of individuals between the anchor and the clicked element.
  - *If* If isShift=true and an anchor exists Then all items between the anchor and the target are added to _selectedIndIds.

- **REQ-IND-033** — When multiple individuals are selected, col. 3 shows a summary with the count of selected items and a 'Delete all selected' button.
  - *If* If _selectedIndIds.size > 1 Then the detail panel shows a multi-selection summary instead of the form.

- **REQ-IND-034** — The delete button (trash icon) in col. 2's header is enabled only when at least one individual is selected.
  - *If* If _selectedIndIds is empty Then ind-del-btn is disabled; otherwise it is enabled.

---

## 6. Drag & Drop

- **REQ-IND-035** — Each individual in the list is draggable onto a class node in col. 1 to change its type.
  - *If* If the user drops an individual onto a class Then its type is updated via PUT /api/individuals/{id}.

- **REQ-IND-036** — During drag, the source element's opacity is reduced to 0.5; the drop target is highlighted with a blue background and outline.
  - *If* If ondragover fires on a class Then background and outline are applied to the target; they are removed on ondragleave.

- **REQ-IND-037** — On drop, if the individual had a type matching the source class (active filter), that type is replaced by the target class; if it had only one type, that type is replaced; otherwise the target class is added.
  - *If* If srcClass and (ind.types includes srcClass) Then types = types.map(t => t===srcClass ? targetClassId : t); else if types.length===1 Then types=[targetClassId]; else types=union(types,[targetClassId]).

- **REQ-IND-038** — Drag & drop is disabled onto owl:Thing (no ondragover/ondrop attributes on that node).
  - *If* If the target is owl:Thing Then dropAttrs() returns an empty string and no drop is processed.

---

## 7. Picker de ressource (ObjectProperty)

- **REQ-IND-039** — Adding a value for an ObjectProperty opens a 'Select Resource' modal with 2 columns (class tree / individual list).
  - *If* If the user clicks ➕ on an ObjectProperty panel Then IndividualEditor.openPicker() is called and the modal is injected into the DOM.

- **REQ-IND-040** — The picker filters displayed classes according to the effective property range (declared range + restriction fillers); if no range, all classes are shown.
  - *If* If effectiveRange is empty Then all classes of the ontology are accessible in the picker.

- **REQ-IND-041** — If the range contains only one class, that class is pre-selected automatically in the picker.
  - *If* If effectiveRange.length === 1 Then pickerSelectClass(effectiveRange[0]) is called on open.

- **REQ-IND-042** — Double-clicking an individual in the picker directly confirms the selection (equivalent to OK).
  - *If* If ondblclick fires on a picker item Then pickerSelectInd() then confirmPicker() are called in sequence.

- **REQ-IND-043** — The picker provides a '＋ New' button to create a new individual directly in the modal without closing it.
  - *If* If the user clicks '＋ New' in the picker Then an inline field appears at the top of the list; confirmation (✓ or Enter) creates the individual via POST and auto-selects it.

- **REQ-IND-044** — In the quick-create picker row, the Escape key cancels input and the Enter key confirms.
  - *If* If key==='Escape' Then the ind-picker-new-row is removed; if key==='Enter' Then _pickerConfirmNew() is called.

- **REQ-IND-045** — Clicking the label of an existing ObjectProperty value in the form navigates to the target individual's record (APP.navigateTo).
  - *If* If the user clicks ind-op-label Then APP.navigateTo('individuals', target) is called.

- **REQ-IND-046** — Clicking a property name (ObjectProperty or DatatypeProperty) in the panel header navigates to that property's definition.
  - *If* If the user clicks cls-frame-tag Then APP.navigateTo('object-properties' or 'datatype-properties', propId) is called.

---

## 8. Propagation des assertions inverses (backend)

- **REQ-IND-047** — On individual creation or update, ObjectProperty assertions are synchronized: if the property has an inverseOf, the inverse assertion is automatically added to the target individual.
  - *If* If a new assertion (property, target) is added and property has inverseOf=inv Then target.objectAssertions receives (inv, ind_id) if not already present.

- **REQ-IND-048** — When an assertion or the individual itself is deleted, the corresponding inverse assertions are removed from target individuals.
  - *If* If an assertion (property, target) is removed and property has inverseOf=inv Then the assertion (inv, ind_id) is removed from target.objectAssertions.

---

## 9. Règles d'affichage (Display Rules)

- **REQ-IND-049** — A ▾ dropdown in col. 1's header gives access to 'Set Display Property' (simple rule) and 'Set Display Properties' (composite rule) options.
  - *If* If the user clicks ▾ Then the ctx-menu appears; it automatically closes on the next click outside the menu.

- **REQ-IND-050** — The simple display rule associates one property (rdfs:label, rdfs:comment, or a DP/OP) with a class to determine the label shown in the list.
  - *If* If a simple display rule is defined for a class Then _resolveDisplayLabel() returns that property's value for each individual of that class.

- **REQ-IND-051** — The composite display rule allows concatenating multiple properties with customizable separators to form the displayed label.
  - *If* If {sep, propId} rows are defined Then _buildMultiLabel() builds the label by concatenating sep+value for each non-empty row.

- **REQ-IND-052** — Display rules are inherited hierarchically: if no rule is defined for a class, the system walks up to parent classes up to __root__ (owl:Thing).
  - *If* If no rule is found for classId Then _getEffectiveDisplayProp() walks up subClassOf until a rule is found or __root__ is reached.

- **REQ-IND-053** — In the simple rule modal, inherited rules are marked '(inherited)' and the class's own rule is marked with ✓.

- **REQ-IND-054** — A 'Clear (use inherited)' button in the modal removes the class's own rule and falls back to the inherited rule.
  - *If* If hasOwn is true Then the Clear button is shown; a click calls setDisplayProp(null) which deletes _displayProps[key].

- **REQ-IND-055** — Display rules are persisted in the backend ontology via PUT /api/display-rules and reloaded from APP.state.ontology.display_rules.
  - *If* If setDisplayProp() or setDisplayPropsMulti() is called Then _saveDisplayRules() sends the update to the backend.

- **REQ-IND-056** — For rdfs:label, label resolution follows the order: requested language → other active languages (Settings.activeLangs) → first available label.
  - *If* If a rdfs:label@xx rule is active Then _getDisplayLabel() looks first for language xx, then active languages, then the first label.

- **REQ-IND-057** — In the individual list, if a display label is resolved, the individual's ID is shown as smaller sub-text.
  - *If* If dispLabel is non-null Then mainText=dispLabel and subText=ind.id; otherwise mainText=ind.id and subText=''.
